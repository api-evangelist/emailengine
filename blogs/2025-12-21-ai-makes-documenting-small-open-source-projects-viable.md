<img alt="AI Makes Documenting Small Open Source Projects Viable" src="https://blog.emailengine.app/content/images/2025/12/ai-documentation-feature.svg" /><p>Let&apos;s be honest about something that every open source maintainer knows but rarely admits: writing documentation is a chore. We love writing code. We love solving problems. We love that dopamine hit when the tests pass. But documentation? That&apos;s the broccoli of software development.</p><p>I&apos;ve been publishing open source projects for over 15 years now. In that time, I&apos;ve created dozens of packages, some with hundreds of millions of downloads. And in hindsight, what a fool I was when it came to docs. At most, I&apos;d write a barely usable README that covers the API basics&#x2014;enough to get someone started, but nowhere near comprehensive. The kind of documentation where you&apos;re constantly switching between the README and the source code, trying to figure out what that third parameter actually does.</p><h2 id="the-documentation-graveyard">The Documentation Graveyard</h2><p>For most of my projects, proper documentation sites simply didn&apos;t exist. The exceptions? Projects that got big enough that the pressure became unbearable. <a href="https://nodemailer.com/?ref=blog.emailengine.app">Nodemailer</a>, which now sees more than a million downloads per day, eventually got a proper documentation site. But that took years of user complaints and questions that could have been answered with better docs.</p><p>The problem isn&apos;t laziness&#x2014;it&apos;s economics. For a small open source project that maybe a few hundred people use, spending days building a documentation site feels like a poor investment. The math just doesn&apos;t work out. So we write a README, promise ourselves we&apos;ll improve it later, and move on to the next interesting problem.</p><p>This leaves the open source ecosystem littered with useful tools wrapped in confusing or incomplete documentation. How many excellent libraries have you abandoned because you couldn&apos;t figure out how to use them properly?</p><h2 id="ai-changes-the-equation">AI Changes the Equation</h2><p>Here&apos;s where things get interesting. With AI tools like Claude, we can now generate comprehensive documentation directly from source code. Not placeholder text or generic templates&#x2014;actual, useful documentation that understands what your code does.</p><p>I created a custom command for Claude Code that does exactly this. You can find it <a href="https://gist.githubusercontent.com/andris9/161b2ea63b10f27553fc304404f392f5/raw/7120f4fccf1fc1f3556ed2c6f94901afa6a2da12/create-docs-site.md?ref=blog.emailengine.app">in this gist</a>. Save it to your Claude Code commands folder, navigate to any project directory, run <code>/create-docs-site</code>, and watch it work.</p><p>The command analyzes your source code, extracts API details, understands the relationships between components, and generates a complete Docusaurus documentation site. We&apos;re talking:</p><ul><li>Getting started guides</li><li>Full API reference</li><li>Usage examples pulled from your tests and example files</li><li>Proper navigation and search</li><li>Even custom SVG graphics for branding</li></ul><h2 id="one-shot-documentation">One-Shot Documentation</h2><p>Here&apos;s a real example: <a href="https://postal-mime.postalsys.com/?ref=blog.emailengine.app">postal-mime.postalsys.com</a>. This entire documentation site was generated in a single run from the project folder. No manual editing, no wrestling with static site generators, no staring at a blank page trying to figure out how to explain what the library does.</p><p>The AI reads your code, understands the intent, and writes documentation that a human would actually find useful. It&apos;s not perfect&#x2014;you might want to add some domain-specific context or adjust the tone&#x2014;but it gets you 90% of the way there in minutes instead of days.</p><h2 id="the-unexpected-benefit">The Unexpected Benefit</h2><p>But here&apos;s what I didn&apos;t anticipate: this approach works for <em>other people&apos;s</em> projects too.</p><p>Ever used a library where you kind of understood the basics but never quite grasped all the options? Where the README shows the happy path but you&apos;re trying to do something slightly different? Now you can just clone the repo, run the documentation generator, and have a full searchable documentation portal explaining everything the library can do.</p><p>You&apos;re not dependent on the maintainer&apos;s documentation skills or available time anymore. If you need better docs for a dependency, you can generate them yourself.</p><h2 id="what-this-means-for-open-source">What This Means for Open Source</h2><p>I think this fundamentally changes the viability calculation for small open source projects. That nifty utility you wrote for yourself and published &quot;just in case someone else needs it&quot;? It can now have professional-grade documentation without the professional-grade time investment.</p><p>The barrier to entry for properly documented open source just dropped by an order of magnitude. And that means more useful tools become actually usable by people who discover them.</p><p>We spent years telling developers they should write better documentation. Turns out, maybe we just needed to wait for the tools that could write it for us.</p>
