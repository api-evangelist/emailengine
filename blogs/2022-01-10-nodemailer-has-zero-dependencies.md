<p>This post is not about <a href="https://emailengine.app/?ref=blog.emailengine.app">EmailEngine</a> but another software project I maintain &#x2013; <a href="https://nodemailer.com/?ref=blog.emailengine.app">Nodemailer</a>. It&apos;s a nifty module for Node.js (as you might assume from the name) that allows to send out emails. And in this post, I&apos;ll explain how and why I ended up having zero dependencies for Nodemailer even though it&apos;s a relatively large project.</p><p>Nodemailer is a long-term project of mine, I started it already 11 years ago and it shows &#x2013; it is downloaded more than 1.5 million times a week, used in a huge number of different projects, and by a lot of companies, both large and small. Any time anyone moves that card on a Trello board and you get an email notification about it? Yup, that&apos;s Nodemailer in action.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img alt="alt" class="kg-image" height="470" src="https://blog.emailengine.app/content/images/2022/01/Screenshot-2022-01-10-at-15.13.30.png" width="1420" /><figcaption><em>_NmP</em> is the Nodemailer&apos;s signature. </code></pre><p>In fact, if you look at the <a href="https://github.com/nodemailer/nodemailer/blob/master/package.json?ref=blog.emailengine.app">package.json</a> of Nodemailer, the &quot;dependency&quot; section is not just empty, it is completely missing.</p><p>So what made me go from that tree-like structure to a single line entry?</p><p>The main reason, install speed.</p><p>Back in 2017 <code>npm install</code> was ridiculously slow. It took forever to properly resolve and download the entire dependency tree. With those v2.7.2 dependencies, it took at least 15 seconds to install Nodemailer, and remember &#x2013; Nodemailer is not a project of its own, it&apos;s a library used by other projects that have many other dependencies. Removing all the dependencies in Nodemailer made that process much faster, install time was reduced to around a single second.</p><p>The other reason was also speed-related but as in development speed. Specifically, release speed. Most of the listed dependencies were managed by myself, these were built for Nodemailer. So for example, if I needed an update in SMTP handling for Nodemailer I changed it in the SMTP handling code by updating <code>smtp-connection</code> module and making a release for it. Then due to fixed dependency numbers in package.json files in other modules, I also had to bump that version number and release newer versions of <code>nodemailer-smtp-pool</code>, <code>nodemailer-smtp-transport</code>, <code>nodemailer-direct-transport</code> and after that, I had to release a new version of Nodemailer itself. Thats&apos; five CHANGELOGs to update and five new versions to publish to <em>npm</em>. Only then this SMTP change came available for the users. Aggregating all modules into a single entity made this pain go away.</p><p>The unexpected upside turned out to be maintainability. This was not a reason to remove dependencies but a result of it. In fact, I wasn&apos;t even able to predict it. After many years Nodemailer today still supports older Node.js versions like v6. I have intentionally not used any newer ES features. This means I use callbacks and not <code>async...await</code> calls. To be honest, Node.js v6 was not so bad at all. It already has the spread operator. Also promises, at least most of the spec. And the arrow function, so need to call the <code>bind(ctx)</code> method. It&apos;s just not modern, and that&apos;s the problem. Most modules nowadays use at least something more modernish and this means these modules do not work with Node v6 as they would trigger <code>SyntaxErrors</code>.</p><p>Supporting older Node.js versions obviously isn&apos;t the only upside regarding maintainability. There are also issues where a dependency update silently breaks something. Known examples are <a href="https://qz.com/646467/how-one-programmer-broke-the-internet-by-deleting-a-tiny-piece-of-code/?ref=blog.emailengine.app">left_pad</a> and <a href="https://snyk.io/blog/open-source-npm-packages-colors-faker/?ref=blog.emailengine.app">colors</a>, but more common issues are smaller and intentional changes, like dropping <code>new</code> by turning a constructor into a regular function or mixing in some newer ES syntax, or moving from CommonJS module system to ESM (looking at you, <a href="https://www.npmjs.com/package/node-fetch?ref=blog.emailengine.app#user-content-commonjs">node-fetch</a>), etc. In any case, once you have bumped that version number in the dependency list, you have to figure out what changed and update your code accordingly. &#xa0;When maintaining Nodemailer I never have to deal with these kinds of things. Because there are no dependencies.</p><p>This did not come without problems.</p><p>While most of the dependencies were actually internal, some were not. I was able to reimplement some things but too large ones I had to drop and figure out alternatives. For example, to use SOCKS proxies that previously were built-in, the current Nodemailer expects you to provide the socks function as part of configuration options. Some features were removed completely and I don&apos;t really miss these.</p><p>The other issue was that these submodules were actually actively used by other people as well, so I couldn&apos;t completely remove these. The main solution was to install Nodemailer as a dependency and then require the needed submodule using a file path.</p><p>For example, previously <code>addressparser</code> could be loaded from a dedicated module:</p><pre><code>const addressparser = require(&apos;addressparser&apos;);
</code></pre><p>Now it is part of Nodemailer and can be loaded from a path even if Nodemailer does not directly expose it:</p><pre><code>const addressparser = require(&apos;nodemailer/lib/addressparser&apos;);
</code></pre><p>In the end, I&apos;m pretty happy with that setup. Nodemailer is pretty much in a frozen state, so I try to not add any new features. From time to time there is some things to fix or some things to change in order to keep compatibility with newer Node.js versions (like switching from <code>new Buffer</code> to <code>Buffer from</code> a few years ago). Having no external parties to depend on, makes that process way easier.</p>
