<img alt="Debugging Webhooks in EmailEngine" src="https://blog.emailengine.app/content/images/2025/05/ChatGPT-Image-5.-mai-2025--11_40_28.png" /><p>Webhooks are the mechanism EmailEngine uses to push account and message events to your application in real time. If you are not receiving the expected notifications, follow the diagnostic flow below to identify the bottleneck.</p><h2 id="1-prove-that-emailengine-can-reach-an-external-endpoint">1. Prove that EmailEngine can reach an external endpoint</h2><p>Start with the simplest external listener so you can observe the raw HTTP requests.</p><blockquote><strong>Example listener</strong><br /><a href="https://webhook.site/?ref=blog.emailengine.app">https://webhook.site/</a> gives you a temporary URL and a live log of incoming requests.<br /><strong>Postal Systems&#xa0;O&#xdc; is not affiliated with webhook.site; we mention it only as a convenient example.</strong><br />Any comparable service&#x2014;e.g. <strong>requestbin.com</strong>, <strong>pipelines.dev</strong>, or a self&#x2011;hosted <strong>ngrok</strong> tunnel&#x2014;works just as well.</blockquote><ol><li>Open your chosen listener and copy the unique callback URL.</li><li>In EmailEngine open <strong>Configuration&#xa0;&#x2192;&#xa0;Webhooks</strong>.</li><li>Tick <strong>Enable webhooks</strong>, paste the URL into <strong>Webhook&#xa0;URL</strong>, select <strong>All&#xa0;events</strong>, and save.</li></ol><p>If you trigger any event now, the listener page should show a POST request from EmailEngine. If nothing appears, the problem is network level (firewall, proxy, DNS) or a typo in the URL&#x2014;fix this before proceeding.</p><h2 id="2-preserve-failed-webhook-attempts-for-inspection">2. Preserve failed webhook attempts for inspection</h2><p>By default EmailEngine discards completed and failed jobs immediately, which hides useful error context.</p><ol><li>Go to <strong>Configuration&#xa0;&#x2192;&#xa0;Service</strong> and set <strong>Completed/failed queue entries to keep</strong> to <strong>100</strong> (or any non&#x2011;zero value).</li><li>Open <strong>Tools&#xa0;&#x2192;&#xa0;Bull&#xa0;Board&#xa0;&#x2192;&#xa0;Webhooks queue</strong>.<ul><li><strong>Delayed</strong> tab &#x2013; jobs that failed but will be retried automatically.</li><li><strong>Failed</strong> tab &#x2013; jobs that exceeded the retry limit.</li></ul></li></ol><p>Open a job to see the full error stack, headers, and payload that EmailEngine sent. This usually reveals TLS errors, 4xx/5xx responses from your server, or JSON parse failures.</p><h2 id="3-verify-that-the-event-itself-exists">3. Verify that the event itself exists</h2><p>Add a fresh mailbox to EmailEngine and wait until its status is <strong>connected</strong>. The listener should immediately receive an <strong>Account&#xa0;initialized</strong> event. If you see it, outbound delivery is working.</p><p>Next, send an email <em>from an external mailbox</em> to the new account. Within 10&#x2011;60&#x202f;s you should receive a <strong>New&#xa0;message</strong> webhook. If not, continue below.</p><h3 id="31-is-the-message-in-the-mailbox-at-all">3.1 Is the message in the mailbox at all?</h3><p>Log in with a regular mail client. If the message landed in <em>Spam</em> or <em>Other</em> folders, EmailEngine will still detect it, but this quick check rules out delivery issues.</p><h3 id="32-can-emailengine-see-the-message">3.2 Can EmailEngine see the message?</h3><p>Run an API request against your EmailEngine to list messages in the INBOX folder.</p><figure class="kg-card kg-code-card"><pre><code class="language-bash">curl &quot;https://EMAILENGINE.HOST/v1/account/ACCOUNT_ID/messages?path=INBOX&amp;pageSize=5&quot; \
  -H &quot;Authorization: Bearer ACCESS_TOKEN&quot;
</code></pre><figcaption><p><span style="white-space: pre-wrap;">Edit EMAILENGINE.HOST, ACCOUNT_ID and ACCESS_TOKEN values</span></p></figcaption></figure><p>If the received message is missing from the JSON response, the account credentials point to a different mailbox (common with shared accounts) or the OAuth token lacks scope.</p><h2 id="4-special-requirements-for-oauth2-backends">4. Special requirements for OAuth2 backends</h2><p>When <strong>Base&#xa0;scopes</strong> is <strong>API</strong> (Gmail API or MS Graph) instead of <strong>IMAP</strong>, EmailEngine relies on provider&#x2011;specific push channels.</p><h3 id="41-gmail-api-cloud-pubsub">4.1 Gmail API + Cloud&#xa0;Pub/Sub</h3><p>Open <strong>Configuration&#xa0;&#x2192;&#xa0;OAuth2</strong>, choose the Gmail OAuth app, and scroll to <strong>Cloud&#xa0;Pub/Sub configuration</strong>. All three items&#x2014;<em>Topic</em>, <em>Subscription</em>, <em>Gmail bindings</em>&#x2014;must show <strong>Created</strong> in green. Anything else means the Google Cloud service account is missing IAM roles or the Pub/Sub API is disabled.</p><h3 id="42-ms-graph-api-subscriptions">4.2 MS&#xa0;Graph API subscriptions</h3><p>Open <strong>Email&#xa0;Accounts</strong>, select the failing account, and scroll to <strong>Change&#xa0;subscription</strong>. The status must be <strong>Created</strong> and the expiration timestamp must be in the future. If not, ensure the Azure app has <strong>Mail.Send</strong> and <strong>offline_access</strong> scopes, and that incoming notifications can reach your EmailEngine host.</p><p>If a subscription has not been created, then the most common cause is that your EmailEngine instance is not reachable from the Microsoft Graph servers. Microsoft Graph sends HTTPS requests to these endpoints:</p><pre><code>https://EMAILENGINE.HOST/oauth/msg/lifecycle
https://EMAILENGINE.HOST/oauth/msg/notification
</code></pre><p>If Graph cannot reach those URLs, for example, if you are serving plain HTTP, your TLS certificates are invalid, <strong>EMAILENGINE.HOST</strong> does not resolve to your instance (set in <strong>Configuration&#x2192;Service&#x2192;ServiceURL</strong>), or outbound connections are firewalled, then new&#x2011;message webhooks for Graph accounts will fail.</p><p>For IMAP accounts, EmailEngine detects new messages on its own. For API back&#x2011;ends such as Microsoft Graph, EmailEngine depends on the provider&#x2019;s webhooks, so external connectivity is mandatory.</p><h2 id="5-recap">5. Recap</h2><ul><li><strong>No request at the listener?</strong> Check the network path and ensure EmailEngine is allowed to reach the URL.</li><li><strong>Job in Failed tab?</strong> Inspect the stack trace for HTTP/TLS problems.</li><li><strong>Message not visible via API?</strong> Verify mailbox credentials and OAuth scopes.</li><li><strong>Using Gmail/MS&#xa0;Graph?</strong> Confirm Pub/Sub or Graph subscription set&#x2011;up.</li></ul><p>Working through the steps above in order narrows the fault domain quickly and gets your integration back to real&#x2011;time status updates.</p>
